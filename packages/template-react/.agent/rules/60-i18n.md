# i18n

The template ships with **i18next** + **react-i18next** preconfigured
for English (`en`, default) and Simplified Chinese (`zh`).

The bootstrap lives in
[src/shared/i18n/index.ts](../../src/shared/i18n/index.ts) and is
invoked once from [src/main.tsx](../../src/main.tsx).

## Why namespaces (and not one big JSON)

Every visible string used to live in one `locales/<lng>.json`. That
file grew O(features). Worse, the active language's full bundle
shipped on first load even for users that never visited a given
feature.

The current setup splits copy two ways:

1. **Per language** — every locale ships in its own chunk.
2. **Per feature (namespace)** — every feature owns its translation
   file; the bundle is fetched the first time the feature's route
   resolves, in parallel with the page chunk.

Net effect: visiting `/` only downloads `common` + `home`; the
`tasks` namespace stays on the CDN until the user clicks Tasks.

## File layout

```
src/shared/i18n/
├── index.ts                      <- bootstrap, exports loadNamespace()
└── locales/
    ├── en/
    │   └── common.json           <- app-shell copy: nav.*, notFound.*, errors.*
    └── zh/
        └── common.json

src/features/<feature>/
└── i18n/
    ├── en.json                   <- ONLY this feature's keys, no prefix
    └── zh.json
```

The namespace name **is** the feature directory name (`counter`,
`tasks`, `home`, …). The `common` namespace is reserved for
app-shell-wide copy (nav links, 404 page, generic errors).

The `import.meta.glob` resolver in `shared/i18n/index.ts` auto-discovers
every feature `i18n/<lng>.json` — adding a new feature requires zero
edits to shared/.

## Hard rules

- **No user-visible string is hard-coded in JSX or in `aria-*` labels.**
  Every visible string goes through `useTranslation()`.
- **One namespace per feature.** Translation keys inside a feature
  file MUST NOT carry the feature name as a prefix; the namespace
  already provides that scoping.
- **Both `en.json` and `zh.json` MUST exist for every namespace** and
  MUST contain the exact same set of keys. When you add a key in
  one, add it in all. (CI may enforce this later via
  `i18next-parser`.)
- Internal error messages, log messages, and developer-only debug
  strings (`throw new Error(...)`) MAY remain in English plain text
  — they are not user-visible.

## Key shape

- Keys are dotted within a namespace. Two levels is typical, three
  is the practical maximum:

  ```jsonc
  // src/features/tasks/i18n/en.json
  {
    "index": { "title": "Tasks", "empty": "No tasks yet — create the first one." },
    "status": { "pending": "Pending", "completed": "Completed" }
  }
  ```

- Leaf segments use **camelCase**: `toastIncreased`, `notFoundDescription`.
- Avoid `text1`, `label`, `value`. Describe intent.
- Interpolated values use double-brace placeholders **inside the
  value**, not in the key: `"Current value: {{value}}"`.

## Consuming keys

Bind the component to the namespace it reads from:

```tsx
import { useTranslation } from 'react-i18next';

function CounterPage() {
  const { t } = useTranslation('counter');
  return <h1>{t('title')}</h1>;        // ← no `counter.` prefix
}
```

The app-shell uses the default `common` namespace:

```tsx
const { t } = useTranslation();        // → defaults to 'common'
t('nav.home')                          // → 'Home'
t('notFound.title')                    // → 'Page not found'
```

A missing key in any feature namespace falls back to `common`
(`fallbackNS: 'common'`), then to the literal key string.

## Lazy-loading contract

Feature namespaces are pulled on demand. Two equally-valid triggers:

1. **Explicit (preferred for route-level pages).** In the feature's
   `routes.tsx`, prefetch the namespace in parallel with the page
   chunk:

   ```tsx
   import { loadNamespace } from '@/shared/i18n';

   const CounterPage = lazy(async () => {
     const [mod] = await Promise.all([
       import('./pages/CounterPage'),
       loadNamespace('counter'),
     ]);
     return { default: mod.CounterPage };
   });
   ```

   By the time the page mounts, both its JS and its translations are
   resident — no flash of fallback keys.

2. **Implicit (works for one-off `useTranslation('foo')` outside the
   page tree).** Skipping the explicit prefetch is OK: react-i18next
   suspends the render until the namespace is loaded, and the shared
   `<Suspense>` in [RootLayout](../../src/app/layouts/RootLayout.tsx)
   catches it. The user sees the route fallback for a few hundred ms
   longer.

Production uses `useSuspense: true`. Tests use `useSuspense: false`
with every namespace pre-bundled (see
[__tests__/setup.ts](../../__tests__/setup.ts)), so component tests
never need `act(...)` around lazy resolves.

## Plurals and gender

i18next handles plurals via suffixes (`_one`, `_other`). Only
introduce them when actually needed; do not preemptively suffix every
key.

## Do not

- **Do not** introduce a new i18n library. Stay on i18next.
- **Do not** place translation strings inside component files as
  constants — they must live in `<ns>/<lng>.json`.
- **Do not** concatenate translated strings (`t('a') + ' ' + t('b')`).
  Compose meaning at the i18n layer with a single key + interpolation.
- **Do not** put a feature's keys into `common`. Keep `common` to
  app-shell copy only.
- **Do not** call `useTranslation()` without an `ns` argument inside
  a feature page — falling back to `common` works mechanically but
  hides which namespace this component depends on. Pass the explicit
  ns argument.
