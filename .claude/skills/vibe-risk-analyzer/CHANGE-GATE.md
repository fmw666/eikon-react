# Change Gate（变更门控）— v1

**适用范围**：对本 skill 任何文件的任何修改（含提案落地、含本文件自身）。
不过门的改动一律不做——宁可少而对,不可多而杂。

---

## Gate 1 — 真实价值（三选一,且必须附当场证据）

| 准入理由 | 证据要求 |
|---|---|
| 修正**已观察到**的评分失真（误报/漏报/可博弈） | 哪次运行、哪个子项、错在哪、正确判定应是什么 |
| 接住**已发生**的工具/流程断点 | 失效命令与输出、现有 fallback 为何没接住 |
| 覆盖**已出现**的真实风险形态（现有维度装不下） | 具体风险实例 + 为何归入任何现有维度都会失真 |

❌ 不过门：「理论上更好」「顺手加的」「别的工具也有这功能」「措辞润色」（错别字/断链修复除外,免门控）。

## Gate 2 — 通用性（单体项目无关,硬性）

- 规范性条文（规则/阈值/画像/schema/模板）**不得**包含：具体项目名、目录名、
  框架宿主词汇（如以 "renderer" 定义画像）、项目私有概念（某项目的章节号/契约名）。
- 单项目特例的正确去处：该项目的 `.vibe-riskrc.json` / 台账（`kind: policy`）/ AGENTS.md——不是 skill 本体。
- 示例**可以**有具体味道,但必须显然可替换（用 `<package>`、`<path/to/file>` 占位优先）。
- 阈值/规则若只有**单一项目**的证据：先进 SELF-EVOLUTION 候选区观察,
  **≥2 个项目复现**或证明属纯逻辑缺陷后才可改规范文。
- 落地后必跑泄漏审计（命中规范文即打回）：

```bash
grep -rn "<项目名>|<专有目录>|<私有概念>" SKILL.md scoring-reference.md \
  report-template.md remediation-prompt.md references/
```

- 会被逐字复制的模板（report-template / remediation-prompt 的模板体）保持英文基底 + 占位符,语言由 agent 填充时决定。

## Gate 3 — 反臃肿

- **净增预算**：单次提案对规范文件合计净增 ≤ 150 行;超出 → 拆分提案或先瘦身等量旧文。
- **one-in-one-out 倾向**：新增规则/子项前,先回答「能否替换或合并一条现有条目?」并在提案里写明结论。
- **新文件门槛**：必须证明无法并入现有文件（职责正交 + 被多处引用）。
- **软上限**（超限的提案必须包含等量瘦身项;上限 = 设立时现状 + 有界余量,
  只约束未来增长,大版本重写时向下棘轮）：SKILL.md ≤ 550 行;scoring-reference ≤ 950;
  report-template ≤ 900;automated-scanning ≤ 560;本文件与 SELF-EVOLUTION 各 ≤ 120。
- 新增维度/子项必须**可自动化**,或有明确的人工裁定步骤与 disputed 出口。
- 脚本必须 `node --check` 通过 + 在 ≥1 个真实仓库冒烟,结果写入提案。

## Gate 4 — 可验证性与兼容

- 规则改动必须落成确定性表述（子项 ID / 公式 / 验收方式）,不得新增自由裁量空间。
- 任何改变分数语义的变更：bump rubric 版本、CHANGELOG 写迁移说明、历史条目标注不可比。

## Gate 5 — 流程

1. 一切变更（含本文件）经提案（PROPOSAL-*.md 或 CHANGELOG 草稿条目均可,小变更可即席提案）。
2. 提案内**逐项勾选**下方清单;任何一项 ❌ → 不落地。
3. 落地后：CHANGELOG 条目 + Gate 2 泄漏审计 + Gate 3 行数核对。

## 提案自查清单（复制进提案）

```
[ ] G1 价值：准入理由=____;证据=____（运行/命令/输出可指认）
[ ] G2 通用：规范文零专名;单项目特例已分流;泄漏 grep 通过;单项目阈值证据已进候选区而非规范文
[ ] G3 体量：净增 ___ 行(≤150);one-in-one-out 结论=____;无超软上限文件(或含瘦身项)
[ ] G4 验证：新/改规则有 ID+公式/验收;分数语义变化已 bump rubric
[ ] G5 流程：CHANGELOG 已拟;脚本 node --check+冒烟(如适用)
```
